# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут задавать значение температуры в системе отопления через конкретное устройство управления в режиме реального времени;
- Пользователи могут включать/выключать отопление 
- Система поддерживает подключение новых устройств для управления отоплением (для этого требуется вызов специалиста с соответствующими правами);
- Система поддерживает отключение устройств для управления отоплением (для этого требуется вызов специалиста с соответствующими правами);
- Система поддерживает обновление информации об устройстве для управления отоплением (для этого требуется вызов специалиста с соответствующими правами). Может быть обновлена информация об имени устройства (name), его типе (type), месте расположения (location), текущем значении (value), единицах измерения (unit), текущем статусе устройства (status);
- Система поддерживает возможность получения актуальных сведений обо всех устройствах. Сведения включают в себя информацию о текущей температуре (Value), актуальном статусе (Status) - активно/неактивно устройтсво, а также о дате последнего изменения в состоянии устройства (LastUpdated);
- Система поддерживает возможность получения актуальных сведений о конкретном устройстве. Сведения включают в себя информацию о текущей температуре (Value), актуальном статусе (Status) - активно/неактивно устройтсво, а также о дате последнего изменения в состоянии устройства (LastUpdated);
- Система поддерживает возможность изменения значения температуры (Value), и текущего статуса (Status) в системе отопления через конкретное устройство управления.

**Мониторинг температуры:**

- Пользователи могут запрашивать актуальную информацию о температуре в отопительной системе с устройства управления, находящегося в определенном месте (локации)
- Система поддерживае возможность получения сведений о системе отопления, находящихся в определенном месте расположения (location). Сведения включают в себя информацию о месте расположения (Location), текущем значении температура (Value), единиц измерения (Unit), текущем статусе (Status), времени последнего изменения (Timestamp), дополнительном описании (Description).

### 2. Анализ архитектуры монолитного приложения

Архитектура приложения представляет из себя монолит на Go с СУБД Postgres. Всё синхронно. Никаких асинхронных вызовов, микросервисов и реактивного взаимодействия в системе нет. Всё управление идёт от сервера к датчику. Данные о температуре также получаются через запрос от сервера к датчику.

### 3. Определение доменов и границы контекстов

В текущем решении можно выделить следующие области:
- Домен: управление устройствами
    - поддомен управления отоплением (вкл, выкл, изменение текущего значения)
	- поддомен администрирования устройств управления отоплением (добавление новых, удаление старых, обновление информации, получение общей информации об устройствах и пр.)
- Домен: мониторинг температуры
    - поддомен сбора актуальной информации с устройств управления системой отопления

В явном виде в текущем решении не представлен, но очевидно, с точки зрения бизнес-задачи, подразумевается
- Домен: управление пользователями
    - поддомен управления правами пользователей (определение, может ли пользователь добавлять новые устройста, сейчас это может делать только "администратор" от компании согласно требованиям, хотя в решении явно никакой проверки нет; определение, может ли пользователь запрашивать информацию с устройств, изменять состояние - сейчас это может делать любой пользователь для любых устройств, даже тех, которые с ним фактически не связаны)
	- поддомен администрирование пользователей (добавление, удаление, обновление информации о пользователе, в текущем решении не используется информация о пользователях, хотя с точки зрения бизнес-требований это подразумевается...)

### **4. Проблемы монолитного решения**

- В текущем решении клиенту для добавление нового устройства в систему требуется вызов специалиста (согласно описанию, чисто технически, кажется, что можно и самостоятельно добавлять устройства через API, возможно оно сейчас находится в закрытой сети, тогда нельзя...), это влечет за собой различные накладные расходы в виде согласования графиков, в целом непрозрачность процесса для клиента и пр., что может негативно сказываться на "заключении сделок", и увеличении времени завершения этих сделок. Это проблема скорее текущего решения не как монолита, т.к. добавить возможность пользователю самостоятельно добавлять новые устройства теоретически можно и в рамках монолитного решения. Это просто как проблема текущего решения; 
- При расширении функционала, в будущем, из-за того, что разные бизнес-функции будут находиться в одном решении-монолите, повышается вероятность возникновения непредсказуемых ошибок. Что-то было изменено в одном месте, это вызвало проблему в другом месте, требуется тратить дополнительнные ресурсы на регрессионное тестирование всего приложения, вместо тестирования конкретной доработки, относящейся к небольшому изолированному микросервису;
- В текущей постановке предлагается добавить 3 новых вида устройств, если продолжить их развитие в рамках монолита, на данном этапе это может быть и не критично, если типов устройств станет сотни, или тысячи, и у каждого будет специфический функционал, поддерживать решение будет очень трудоемко. Команда, отвечающая за одни виды устройств начнет мешать команде, отвечающей за другие виды устройств, и возникнет хаос, который увеличит стоимость поддержки очень сильно, а эффективность разработки с такой же силой понизит;
- Увеличивается Time to market при росте решения, т.к. увеличивается время на тестирование всего решения (чтобы убедиться, что не задели ничего лишнего при внедрении новой функциональности). Также это негативно сказывается на пользовательском опыте (выпуск исправления ошибок также сильно замедляется)
- Трудность в масштабировании отдельных компонентов системы. Т.к. все решение деплоится целиком, в случае, если возросла нагрузка только на один вид устройств, невозможно будет увеличить число экземпляров сервисов, работающих с этим типом устройств, придется либо попробовать концентированно оптимизировать блок кода, относящийся к проблемному месту (но регрессионное тестирование все равно потребуется всего приложения), либо создавать дополнительные экземпляры всего большого решения, пытаясь справиться с нагрузкой. В данной ситуации будет использовано больше ресурсов железа, чем нам на самом деле требуется, и будет потрачено больше денег, чем требовалось бы в случае выделенного "нагруженного" микросервиса, что для компании также не очень хорошо;  
- В текущем решении не реализованы сбор метрик жизнеспособности системы (не связано с монолитом, а в целом, как проблема);
- В текущем решении не реализована авторизация, аутентификация пользователей (не связано с монолитов, а в целом, как проблема);
- Высокий уровень поломки всего приложения при обновлениях.  Если были внесены ломающие правки, и они ушли в промышленную эксплуатацию, сломается все приложение целиком, в случае разделения ответственности, и раздельный деплой разных функциональностей-микросервисов, отказоустойчивость возрастает, сломается только одна небольшая бизнес-функциональность, само приложение в целом продолжит работать.

# Задание 5. Работа с docker и docker-compose

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

1) сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```

2) Приложение следует упаковать в Docker и добавить в docker-compose. Порт по умолчанию должен быть 8081

3) Кроме того для smart_home приложения требуется база данных - добавьте в docker-compose файл настройки для запуска postgres с указанием скрипта инициализации ./smart_home/init.sql

Для проверки можно использовать Postman коллекцию smarthome-api.postman_collection.json и вызвать:

- Create Sensor
- Get All Sensors

Должно при каждом вызове отображаться разное значение температуры

Ревьюер будет проверять точно так же.
